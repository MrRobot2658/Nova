# Nova 技术架构

> 把 Hermes Agent 包装成 RPA 替代品。六层架构，28+ 预制 Skill，自然语言驱动。

## 六层详解

### L1 — CLI & 安装器

| 组件 | 说明 |
|------|------|
| `nova` CLI | 统一的命令行入口，`nova "帮我做XXX"` |
| `install.sh` | 一键安装：检测环境→安装依赖→拉取 Skill→验证 |
| `healthcheck.sh` | 环境健康检查：Python/Node/Git/网络/API Key 全部校验 |

**设计决策**：不是给开发者用的 CLI 工具，而是给业务人员用的「一句话终端」。跟传统 RPA 的控制台是同一层概念，但交互方式从图形化拖拽变成了自然语言。

### L2 — Hermes Agent Core

Nova 不自己造 Agent 引擎，直接跑在 Hermes 之上。

| 模块 | 职责 |
|------|------|
| **自然语言解析** | 用户说「汇总飞书文档」，解析出目标平台（飞书）、动作（汇总）、输出（Excel） |
| **意图路由** | 匹配最合适的 Skill 组合，一个复杂需求可能触发 3-5 个 Skill 协同 |
| **上下文记忆** | 跨轮对话记住用户偏好、历史操作、数据中间结果 |
| **工具调度** | 把 Skill 按顺序执行，处理返回值，出错自动重试或换方案 |
| **模型路由** | 简单任务→MiniMax（快/省），复杂推理→DeepSeek V4，视觉→QwenVL，故障切换自动降级 |

**为什么不做独立引擎**：Hermes 已经解决了 Agent 的基础设施——会话管理、工具注册、多模型路由、持久化记忆。Nova 的价值在上层的 Skill 封装和分发，不在底层引擎。

### L3 — Skill 系统 (28 个预制 Skill)

Nova 的 Slide 不是零散的脚本，而是按场景分组的标准化能力单元。每个 Skill 都有统一的接口规范（输入/输出/错误处理/超时）。

**分组逻辑**：不是按技术栈分，而是按「用户想干什么」分。

**扩展机制**：新 Skill 按 Hermes 标准格式编写 SKILL.md，放入 skills 目录即自动注册。用户也可以写自己的 Skill。

### L4 — AI 引擎

Nova 不是简单的脚本串联，AI 引擎是核心差异化。

| 能力 | 实现 | 价值 |
|------|------|------|
| **LLM 推理** | DeepSeek V4（主力）/ MiniMax M2.7（快速）/ Qwen（容灾） | 理解非结构化内容、做决策、生成内容 |
| **视觉理解** | QwenVL OCR | 看懂图片/截图/PDF |
| **语音合成** | Edge TTS / OpenAI TTS | 文本转语音输出 |
| **网络代理** | HTTP/SOCKS5 代理层 | 国内外网络环境适配 |

**跟传统 RPA 的本质区别**：RPA 只能处理「这个按钮在第 3 行第 5 列」这种确定性逻辑。Nova 的 AI 引擎能理解「这封邮件的语气有点急，帮我起草一个安抚回复」——这是非确定性推理，传统 RPA 做不到。

### L5 — Runtime & 调度

自动化不是一次性操作，是持续运行的。Nova 的 Runtime 层负责把一次性指令变成长期运行的自动化流程。

| 机制 | 触发方式 | 典型场景 |
|------|---------|---------|
| **cronjob** | 时间驱动（每天 9:00 / 每小时 / 每周末） | 日报、周报、定期巡检 |
| **webhook** | 事件驱动（表单提交 / Git push / API 回调） | 审批流、CI/CD 通知 |
| **computer_use** | 对话驱动（用户说→就执行） | 桌面操作、GUI 自动化 |
| **browser-act** | 对话驱动 + 定时 | 网页采集、表单填写 |

**关键设计**：所有定时/事件任务在 Hermes 的 cron 引擎上运行，每个任务独立 session，互不干扰。支持失败重试、超时控制、执行日志。

### L6 — 集成目标

Nova 不是孤岛，它的价值在于打通已有工具。

```
┌─────────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 飞书/企业微信 │  │  钉钉    │  │  Google  │  │  Notion  │  │ Airtable │
│   /钉钉      │  │          │  │Workspace │  │          │  │          │
└──────┬───────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
       └───────────────┴────────────┴────────────┴──────────────┘
                           │
                    ┌──────▼──────┐
                    │    Nova     │
                    └──────┬──────┘
                           │
       ┌───────────────────┼───────────────────┐
       │                   │                   │
┌──────▼──────┐  ┌─────────▼────┐  ┌───────────▼──┐
│ 公众号/小红书 │  │  邮件/IM     │  │  macOS 桌面   │
```

核心逻辑：**数据从任意入口进来 → Nova 处理 → 输出到任意目标**。飞书写的内容可以自动发公众号，邮件收到的 PDF 可以自动解析进 Notion，微信消息可以触发一条自动化流程。

## 与传统 RPA 架构对比

| 层面 | 传统 RPA (UiPath/影刀) | Nova |
|------|----------------------|------|
| **交互层** | 可视化拖拽设计器 | 自然语言 CLI |
| **逻辑层** | 流程图/状态机（手动编排） | AI 自主推理 + Skill 组合 |
| **执行层** | 选择器 + 脚本引擎 | Hermes Agent + 工具调度 |
| **数据层** | 结构化变量 + Excel | AI 原生理解（结构化+非结构化） |
| **调度层** | 内置 Orchestrator | Hermes cronjob + webhook |
| **集成层** | 预置连接器（有限） | 任意 API / 网页 / 桌面软件 |

**Nova 的架构优势**：
1. **横向扩展零成本**——加一个新平台就是写一个 Skill，不需要学新平台
2. **容错是内置的**——传统 RPA 选择器崩了整个流程挂，Nova 出错后 AI 自己想办法
3. **非结构化是原生能力**——不是靠 OCR 硬凑，AI 天然理解文档/邮件/网页
4. **维护成本极低**——改需求就是改一句话，不用重新画流程图

## 技术选型依据

| 选择 | 理由 |
|------|------|
| **基于 Hermes 而非自研引擎** | Hermes 已解决 Agent 基础设施，Nova 聚焦 Skill 封装和分发 |
| **DeepSeek V4 为主模型** | 中文推理最强、价格合理、开源可控 |
| **macOS 优先** | 目标用户是企业老板/管理者，Mac 是标配。Windows 版可通过 WSL2 跑 |
| **Skill 而非 Plugin** | Skill 是自包含的指令+代码单元，Plugin 是外部依赖。Skill 更可控、更好分发 |
| **cronjob 而非独立调度器** | 定时任务是 Hermes 内置能力，复用而非另起炉灶 |
