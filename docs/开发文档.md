## 1. 环境准备

| 依赖 | 版本/说明 |
|------|----------|
| 操作系统 | macOS（一等支持）；Windows 走 WSL2（Ubuntu 22.04） |
| Hermes Agent | 必装，位于 `~/.hermes`；Nova 是其上层封装 |
| Homebrew | macOS 包管理（安装脚本会校验） |
| Python | 3.x（`yt-dlp`、`python-dotenv`、`requests` 等） |
| Node.js | ≥ 18（官网本地预览 / 工具链）；当前环境 v24 |
| Git | 版本管理 |
| Vercel CLI | 官网部署（`brew install vercel` 或 `npm i -g vercel`） |

依赖由安装脚本自动处理（见 §4）。

## 3. 配置

### 3.1 环境变量

```bash
cp .env.example .env      # 然后编辑 .env 填入真实密钥
```

`.env` 已被 `.gitignore` 忽略，**不会被提交**。各变量含义见 `.env.example` 注释。

### 3.2 模型配置（当前：仅 DeepSeek）

当前阶段统一使用 **DeepSeek**（OpenAI 兼容协议）。`.env` 关键项：

```ini
DEEPSEEK_API_KEY=sk-********************************
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-pro        # 复杂推理 / 决策（主力）
DEEPSEEK_MODEL_FAST=deepseek-v4-flash # 快速 / 低成本任务
```

可用模型（`GET /models` 实测）：

| 模型 id | 定位 | 用途 |
|---------|------|------|
| `deepseek-v4-pro` | 强推理 | 意图解析、决策、风险判断、内容生成（默认） |
| `deepseek-v4-flash` | 快/省 | 分类、抽取、改写等高频轻任务 |

> 架构中规划的 MiniMax（快）/ QwenVL（视觉）/ Qwen（容灾）暂不启用；接入时只需在 `.env` 补充对应 key，由 AI 引擎模块（[详细设计 §6](详细设计.md#6-m5ai-引擎模块)）按任务路由。视觉 OCR 在仅 DeepSeek 阶段先不可用。

### 3.3 在 Hermes 中生效

`nova` 实为 `hermes chat --profile devops` 的薄封装。把上述模型配置写入 Hermes 的 `devops` profile（base_url / api_key / model 指向 DeepSeek），即让 Nova 默认走 DeepSeek。具体字段以 Hermes 配置规范为准；推荐用环境变量注入，避免明文落库。

### 3.4 密钥安全约定（重要）

- **本仓库是 public** —— 严禁把任何真实 key 写入会被提交的文件（源码、文档、`.env.example`、commit message）。
- 真实 key 只放在本地 `.env`（已忽略）或系统环境变量 / Hermes 安全存储。
- 一旦 key 在聊天、截图、日志、PR 中暴露，视为泄露，请在 DeepSeek 控制台 **立即轮换**。
- 提交前自检：`git status` 确认 `.env` 未被跟踪；`git diff --cached | grep -i sk-` 应无命中。

### 3.5 快速验证 key

```bash
set -a; . ./.env; set +a
curl -s https://api.deepseek.com/models \
  -H "Authorization: Bearer $DEEPSEEK_API_KEY" -w "\nHTTP %{http_code}\n"
# 期望 HTTP 200，并列出 deepseek-v4-pro / deepseek-v4-flash
```

## 4. 安装与运行

### 4.1 安装

```bash
bash code/install.sh          # macOS：装依赖 + 写入 ~/bin/nova + 校验 ~/.hermes
bash code/install.sh --dmg    # 另外打包 Nova-Installer.dmg
# Windows：powershell -ExecutionPolicy Bypass -File code/scripts/install-windows.ps1
```

### 4.2 运行

```bash
nova "帮我把这周飞书文档汇总成 Excel 发给老板"
```

`nova` → `cd ~ && hermes chat --profile devops -- "$@"`。请求生命周期（CLI → 意图解析 → 编排 → Skill → 输出）见[概要设计 §5.1](概要设计.md#51-核心请求生命周期对话驱动)。

### 4.3 官网本地预览

```bash
python3 -m http.server 8000 --directory site   # 或 npx serve site
# 打开 http://localhost:8000
```

## 5. Skill 开发

Skill 是 Nova 的能力载体。新增能力 = 新增一个 Skill，不改核心（见[详细设计 §5](详细设计.md#5-m4skill-系统模块)）。

### 5.1 目录与 SKILL.md 规范

```
skills/<skill-name>/
├── SKILL.md        # 元信息（必需）
└── （实现脚本/资源）
```

`SKILL.md` 约定字段：

| 字段 | 说明 |
|------|------|
| `name` | 唯一标识（kebab-case） |
| `description` | 一句话能力描述（用于意图路由匹配） |
| `group` | 分组：office / data / engine / publish |
| `inputs` | 入参 schema |
| `outputs` | 出参 schema |
| `errors` | 可能错误码（见 §5.4） |
| `timeout` | 超时秒数 |

### 5.2 新建一个 Skill

1. 在 `code/skills/` 下建目录与 `SKILL.md`。
2. 实现逻辑，遵守统一调用协议（§5.3）。
3. 放入目录即自动注册（启动扫描），无需改路由代码。
4. 自测：用一句会命中该 Skill 的自然语言指令跑通。

### 5.3 调用协议（统一返回）

```jsonc
// 入参
{ "skill": "feishu-doc-create", "params": { ... }, "context": { ... } }
// 成功
{ "ok": true, "data": { ... }, "artifacts": [ ... ], "costMs": 1830 }
// 失败
{ "ok": false, "error": { "code": "AUTH_EXPIRED", "msg": "...", "retryable": true } }
```

- 需要模型时，向 AI 引擎声明**任务类型**（复杂/快速/视觉），由其选模型，不在 Skill 内写死 model id。
- 操作 GUI/浏览器走自动化执行模块；读写平台走集成适配模块。

### 5.4 错误码分级

| 级别 | 含义 | 默认处理 |
|------|------|---------|
| `INPUT_*` | 入参/意图不全 | 回问或上下文补全，不重试 |
| `AUTH_*` | 凭证缺失/过期 | 触发重新授权 |
| `RATE_*` / `NET_*` | 限流/网络 | 指数退避重试（默认 2 次） |
| `RUNTIME_*` | 执行期错误 | 重试，超限降级 |
| `FATAL_*` | 不可恢复 | 终止当前步并上报 |

Skill 只返回标准错误码，跨步重试/降级由编排调度模块决定（[详细设计 §4.6](详细设计.md#46-错误处理与降级)）。

## 6. 调试与日志

- 每次执行记录：意图（Intent）、编排计划（Plan）、各 Skill 耗时与状态。
- 定时/事件任务在独立 Session 运行并保留执行历史，便于回溯（[详细设计 §7](详细设计.md#7-m6runtime-调度模块)）。
- 调试模型层：先用 §3.5 验证 key 与连通性，再排查上层。

## 7. 官网开发与部署

- 站点在 `site/`：纯静态 HTML/CSS/JS，无构建步骤、无依赖。
  - `index.html` 落地页 + 购买；`demo.html` + `assets/demo.js` 交互式 Demo；`assets/style.css` 品牌样式。
- 本地预览见 §4.3。
- 部署：根目录 `vercel.json`（`outputDirectory: site`，无 build）。Vercel 项目已 git 连接，**push 到 `main` 自动部署到生产**；如需手动：`vercel deploy --prod`。
- 线上：<https://nova-nine-peach.vercel.app>。

## 8. 提交与协作规范

| 项 | 约定 |
|----|------|
| 分支 | 不直接在 `main` 改大功能；开 `feat/*`、`docs/*`、`fix/*` 分支，再合并 |
| Commit message | 用前缀 `feat: / fix: / docs: / chore:`；正文说清「做了什么、为什么」 |
| 署名 | commit 末尾保留 `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`（本仓库约定，见 CLAUDE.md） |
| 文档一致性 | Skill 数（28 = 13/5/4/2）、六层架构、模型路由、竞品对标（仅 UiPath/影刀）在多处出现，改一处要同步其余（清单见 `CLAUDE.md`） |
| 提交前自检 | `.env` 未被跟踪；diff 中无 `sk-` 密钥；中文文档语气与既有一致 |

## 9. 测试与发布

- **现状**：本仓库以脚本 + 文档 + 静态站为主，无自动化测试框架。
- **手动验证清单**：
  1. `bash code/install.sh` 能跑通依赖安装与 `~/bin/nova` 写入。
  2. §3.5 的 key 验证返回 200。
  3. 至少一条端到端指令（如「汇总飞书文档」）跑通主链路。
  4. 官网本地预览三页面（首页/购买弹窗/Demo）正常。
- **发布**：
  - 客户端：`bash code/install.sh --dmg` 产出 `.dmg`。
  - 官网：push `main` → Vercel 自动上线。

## 10. FAQ

| 问题 | 处理 |
|------|------|
| `nova` 命令找不到 | 确认 `~/bin` 在 `PATH`，且 `install.sh` 已执行 |
| 提示未检测到 Hermes | 先安装 Hermes 到 `~/.hermes`，再跑 `install.sh` |
| 模型 401 | `.env` 的 `DEEPSEEK_API_KEY` 失效或被轮换，更新后重试 |
| 视觉/OCR 不可用 | 仅 DeepSeek 阶段未接入 QwenVL，按 §3.2 后续启用 |
| 官网改了没生效 | 确认已 push 到 `main`，等待 Vercel 部署完成（约数十秒） |
