## 1. 架构总览

### 1.1 形态定位

Nova 的正式交付形态是一个 **Electron 桌面客户端**：

- **桌面端用 TypeScript 实现，框架用 Electron**；UI 风格参考 [Manus](https://manus.im)。
- **安装包内置经过二次开发的 Hermes 服务端**——把整个 Hermes 源代码纳入仓库、在其上做二开，并随安装包一起分发。用户双击安装即用，**无需单独安装 Hermes**。
- 早期的 `nova` CLI（`hermes chat` 薄封装）保留为调试/无界面入口，不再是主要形态。

### 1.2 总体拓扑

```
┌─ Nova.app  (Electron 桌面端, TypeScript) ────────────────────┐
│                                                              │
│  Renderer (TS + React) ── Manus 风格 UI                       │
│    会话/任务侧栏 · 对话区 · 「执行/计算机」面板 · 结果预览       │
│            ▲  contextBridge 暴露的安全 API (window.nova.*)     │
│  Preload (安全桥, contextIsolation)                           │
│            ▲  ipcMain / ipcRenderer                           │
│  Main (Node/TS)                                              │
│    窗口/生命周期 · 托盘/快捷键 · IPC 路由                       │
│    └─ 启动并监管「内置 Hermes」子进程（spawn/健康检查/重启）     │
│            │  本地通信 (localhost HTTP 或 stdio)               │
│            ▼                                                  │
│  ┌─ 内置 Hermes 服务端 (fork 二开, sidecar 子进程) ──────────┐ │
│  │  Agent 核心：意图解析 · 编排调度 · 记忆 · 工具调度          │ │
│  │  Skill 系统 (28, 4 组) · Runtime 调度 (cronjob/webhook)    │ │
│  └────────────────────────────────────────────────────────┘ │
└───────────────────────────┬──────────────────────────────────┘
                            │ 出站
        DeepSeek API (deepseek-v4-pro / -flash) · 各集成平台 API
```

### 1.3 技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 桌面外壳 | **Electron** | 跨平台桌面；Main/Renderer 双进程 |
| 语言 | **TypeScript** | main / preload / renderer 全量 TS |
| UI 框架 | **React**（默认） | 配 Tailwind CSS / CSS Modules；样式参考 Manus |
| 状态管理 | Zustand 或 Redux Toolkit（按需） | 渲染层会话/任务状态 |
| 进程通信 | Electron IPC + `contextBridge` | 渲染层不直连 Node |
| 后端 | **内置 Hermes 服务端（fork 二开）** | 以 sidecar 子进程运行 |
| 打包 | **electron-builder** | 产出 .dmg / NSIS .exe，含 Hermes `extraResources` |
| 包管理 | pnpm（默认） | monorepo 友好 |

> React / electron-builder / pnpm 为推荐默认，可按团队偏好替换；架构其余部分不依赖具体选择。

### 1.4 进程模型

| 进程 | 职责 | 安全约束 |
|------|------|---------|
| **Main** | 应用生命周期、窗口、系统集成；**拉起并监管内置 Hermes 子进程**；IPC 路由；本地与 Hermes 通信 | 唯一可访问 Node/FS/子进程 |
| **Preload** | 通过 `contextBridge` 暴露最小化 `window.nova.*` API | `contextIsolation: true`、`nodeIntegration: false` |
| **Renderer** | 纯 UI（React），经 preload API 间接调用能力 | 不直接接触 Node/网络密钥 |
| **Hermes sidecar** | Agent 核心 + Skill + 调度；监听 localhost | 仅本机回环，不对外暴露端口 |

请求链路：UI 输入自然语言 → Renderer →(IPC)→ Main →(localhost)→ Hermes（意图→编排→Skill→执行）→ 结果回传 → UI 渲染。与[概要设计 §5 数据流](概要设计.md#5-模块间接口与数据流)一致，桌面端对应 **L1 接入层**，内置 Hermes 承载 **L2–L6**。

### 1.5 内置 Hermes 与二次开发

- **纳入方式**：整个 Hermes 源代码以 fork/vendored 形式放入 `code/hermes/`，建议用 **git submodule 或 git subtree** 跟踪上游，便于定期 rebase 上游更新。
- **二开内容**：
  - Nova 专属 `devops` profile 与默认行为；
  - Skill 注册/分组适配（[详细设计 §5](详细设计.md#5-m4skill-系统模块)）；
  - 模型路由接入 DeepSeek（见 §3.2）；
  - 与桌面端的本地通信接口（localhost HTTP / stdio）及健康检查协议。
- **改动隔离**：二开尽量以 patch/overlay 形式集中，锁定上游版本，降低跟随成本。

### 1.6 安装包内置 Hermes

- 用 electron-builder 的 `extraResources` 把 Hermes 运行时及其依赖运行时（按 Hermes 实际技术栈，如 Node/Python）一并打入安装包。
- 首次启动由 Main 进程从 `process.resourcesPath` 定位并 spawn Hermes，对用户透明。
- **平台说明**：macOS 产出 `.dmg`（预留签名/公证）。Windows 若 Hermes 无原生构建，仍可经 WSL2 承载 Hermes，由安装器编排（沿用 `code/scripts/install-windows.ps1` 思路）；Hermes 跨平台能力决定最终打包方式（见 §10 风险）。

### 1.7 UI 设计参考 Manus

- **三栏布局**：左 = 会话/任务列表；中 = 对话流（自然语言指令 + Agent 回复）；右 = **「执行/计算机」面板**，实时展示 Skill 调用、浏览器/终端/文件操作过程与产物预览。
- **可观测优先**：强调"看得见 AI 在干什么"——执行时间线与官网 `site/demo.html` 的演示一脉相承，桌面端是其真实版。
- **视觉**：浅色、克制、卡片化。初期先对齐**布局与交互范式**，不照搬 Manus 的具体视觉资产。

## 4. 配置

### 4.1 环境变量

```bash
cp .env.example .env      # 编辑 .env 填入真实密钥
```

`.env` 已被 `.gitignore` 忽略，**不会被提交**。

### 4.2 模型配置（当前：仅 DeepSeek）

```ini
DEEPSEEK_API_KEY=sk-********************************
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-pro        # 复杂推理 / 决策（主力）
DEEPSEEK_MODEL_FAST=deepseek-v4-flash # 快速 / 低成本任务
```

可用模型（`GET /models` 实测）：

| 模型 id | 定位 | 用途 |
|---------|------|------|
| `deepseek-v4-pro` | 强推理 | 意图解析、决策、内容生成（默认） |
| `deepseek-v4-flash` | 快/省 | 分类、抽取、改写等高频轻任务 |

> 内置 Hermes 的 `devops` profile 把 base_url / api_key / model 指向 DeepSeek，即让 Nova 默认走 DeepSeek。MiniMax / QwenVL / Qwen 暂不启用，接入时补 `.env` 对应 key 即可。

### 4.3 密钥安全约定（重要）

- **本仓库是 public** —— 严禁把任何真实 key 写入会被提交的文件（源码、文档、`.env.example`、commit message）。
- 真实 key 只放本地 `.env`（已忽略）或系统环境变量 / Hermes 安全存储；桌面端打包产物中不内嵌明文 key。
- key 一旦在聊天/截图/日志/PR 暴露即视为泄露，请到 DeepSeek 控制台 **立即轮换**。
- 提交前自检：`git status` 确认 `.env` 未跟踪；`git diff --cached | grep -i sk-` 应无命中。

### 4.4 快速验证 key

```bash
set -a; . ./.env; set +a
curl -s https://api.deepseek.com/models \
  -H "Authorization: Bearer $DEEPSEEK_API_KEY" -w "\nHTTP %{http_code}\n"
# 期望 HTTP 200，列出 deepseek-v4-pro / deepseek-v4-flash
```

## 6. Skill 开发

Skill 是 Nova 的能力载体，运行在内置 Hermes 中。新增能力 = 新增一个 Skill，不改核心（[详细设计 §5](详细设计.md#5-m4skill-系统模块)）。

### 6.1 目录与 SKILL.md 规范

```
code/skills/<skill-name>/
├── SKILL.md        # 元信息（必需）
└── （实现脚本/资源）
```

`SKILL.md` 字段：`name` / `description` / `group`(office·data·engine·publish) / `inputs` / `outputs` / `errors` / `timeout`。

### 6.2 统一调用协议

```jsonc
// 入参
{ "skill": "feishu-doc-create", "params": { ... }, "context": { ... } }
// 成功
{ "ok": true, "data": { ... }, "artifacts": [ ... ], "costMs": 1830 }
// 失败
{ "ok": false, "error": { "code": "AUTH_EXPIRED", "msg": "...", "retryable": true } }
```

- 需要模型时声明**任务类型**（复杂/快速），由模型路由选 `deepseek-v4-pro` / `-flash`，不在 Skill 内写死 model id。
- 操作 GUI/浏览器走自动化执行模块；读写平台走集成适配模块。

### 6.3 错误码分级

| 级别 | 含义 | 默认处理 |
|------|------|---------|
| `INPUT_*` | 入参/意图不全 | 回问或上下文补全，不重试 |
| `AUTH_*` | 凭证缺失/过期 | 触发重新授权 |
| `RATE_*` / `NET_*` | 限流/网络 | 指数退避重试（默认 2 次） |
| `RUNTIME_*` | 执行期错误 | 重试，超限降级 |
| `FATAL_*` | 不可恢复 | 终止当前步并上报 |

跨步重试/降级由编排调度模块决定（[详细设计 §4.6](详细设计.md#46-错误处理与降级)）。

## 7. 桌面端开发

### 7.1 三进程分工

- **Main（`src/main`）**：创建 `BrowserWindow`；`app` 生命周期；用 `child_process.spawn` 启动内置 Hermes，做健康检查与退出清理；注册 `ipcMain.handle` 处理来自渲染层的请求并转发给 Hermes。
- **Preload（`src/preload`）**：`contextBridge.exposeInMainWorld('nova', { send, onStream, ... })`，只暴露白名单 API。
- **Renderer（`src/renderer`）**：React 实现 Manus 风格三栏 UI；通过 `window.nova.*` 与 Main 通信，流式渲染执行过程。

### 7.2 与内置 Hermes 通信

```
Renderer ──window.nova.run(text)──▶ Main
Main ──localhost HTTP / stdio──▶ Hermes（意图→编排→Skill）
Hermes ──事件流（每步状态/产物）──▶ Main ──IPC stream──▶ Renderer（实时渲染执行面板）
```

事件流驱动右侧「执行/计算机」面板：每个 Skill 调用对应一条时间线项（进行中 ⟳ → 完成 ✓ / 失败 ✗），与官网 `demo.js` 的渲染模型一致。

### 7.3 UI 模块（Manus 风格）

| 区域 | 组件 | 内容 |
|------|------|------|
| 左栏 | SessionList | 会话/任务历史、新建任务 |
| 中栏 | Conversation | 自然语言输入框 + 对话气泡（用户/Agent） |
| 右栏 | ExecutionPanel | 实时 Skill 执行时间线、浏览器/终端/文件预览、结果产物 |
| 全局 | StatusBar | Hermes 连接状态、当前模型（deepseek-v4-pro） |

## 8. 构建与打包

`electron-builder.yml` 关键点（示意）：

```yaml

    to: hermes
mac: { target: dmg }
```

## 10. 提交规范 / 测试 / 风险

### 10.1 提交与协作

| 项 | 约定 |
|----|------|
| 分支 | 开 `feat/*`、`docs/*`、`fix/*`，再合并 `main`；不在 `main` 直接做大改 |
| Commit | 前缀 `feat:/fix:/docs:/chore:`；正文讲清「做了什么、为什么」 |
| 署名 | 保留 `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>` |
| 一致性 | Skill 数(28=13/5/4/2)、六层架构、模型路由、竞品对标(仅 UiPath/影刀)多处出现，改一处同步其余（清单见 `CLAUDE.md`） |
| 自检 | `.env` 未跟踪；diff 无 `sk-` 密钥 |

### 10.2 测试与手动验证

1. `pnpm dev` 桌面端能启动并成功拉起内置 Hermes（状态栏显示已连接）。
2. §4.4 的 key 验证返回 200。
3. 一条端到端指令跑通：UI 输入 → 执行面板逐步显示 Skill → 产出结果。
4. `pnpm dist` 产物可安装、安装后无需外部 Hermes 即可运行。

### 10.3 关键风险

| 风险 | 对策 |
|------|------|
| Hermes 跨平台（尤其 Windows 原生）能力不足 | Windows 经 WSL2 承载；或推动 Hermes 原生构建 |
| 内置 Hermes 体积大 / 启动慢 | 裁剪运行时依赖；首启动预热 + 进度提示 |
| 上游 Hermes 更新与二开冲突 | submodule 锁版本 + patch/overlay 集中改动，定期 rebase |
| 安装包内嵌密钥风险 | 产物不内嵌 key，运行期从用户配置/安全存储读取 |
| 渲染层安全 | `contextIsolation` 开、`nodeIntegration` 关，IPC 白名单 |

## 11. FAQ

| 问题 | 处理 |
|------|------|
| 桌面端起不来 / 连不上 Hermes | 看 Main 进程日志，确认 Hermes 子进程已 spawn 且健康检查通过 |
| 模型 401 | `.env` 的 `DEEPSEEK_API_KEY` 失效或被轮换，更新后重试 |
| 视觉/OCR 不可用 | 仅 DeepSeek 阶段未接入 QwenVL，按 §4.2 后续启用 |
| 打包后找不到 Hermes | 确认 `extraResources` 已包含 Hermes，且 Main 用 `process.resourcesPath` 定位 |
| 官网改了没生效 | 确认已 push `main`，等 Vercel 部署完成 |
