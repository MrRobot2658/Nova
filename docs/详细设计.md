## 1. 通用约定

### 1.1 Skill 调用协议

所有 Skill 遵循统一协议，便于编排器无差别调度：

```jsonc
// 入参
{ "skill": "feishu-doc-create", "params": { ... }, "context": { ... } }
// 出参（成功）
{ "ok": true, "data": { ... }, "artifacts": [ ... ], "costMs": 1830 }
// 出参（失败）
{ "ok": false, "error": { "code": "AUTH_EXPIRED", "msg": "...", "retryable": true } }
```

### 1.2 错误码分级

| 级别 | 含义 | 默认处理 |
|------|------|---------|
| `INPUT_*` | 入参/意图不完整 | 回问用户或用上下文补全，不重试 |
| `AUTH_*` | 凭证缺失/过期 | 触发重新授权流程，暂停该步 |
| `RATE_*` / `NET_*` | 限流/网络 | 退避重试（指数退避，默认 2 次） |
| `RUNTIME_*` | 执行期错误 | 重试；超阈值则降级换方案 |
| `FATAL_*` | 不可恢复 | 终止当前步，上报并保留已完成结果 |

### 1.3 通用时序符号

下文时序图统一用 `→` 表示调用、`⇢` 表示异步、`✓/✗` 表示成功/失败分支。

## 2. M1·CLI 与安装模块

### 2.1 职责

- 提供统一命令入口 `nova "<自然语言指令>"`。
- 跨平台一键安装：检测环境 → 安装依赖 → 写入 `nova` 封装 → 校验 Hermes。
- macOS 支持 `--dmg` 打包分发；Windows 经 WSL2 部署并创建桌面快捷方式。

### 2.2 组成

| 组件 | 文件 | 说明 |
|------|------|------|
| `nova` 封装 | `~/bin/nova`（安装时生成） | `cd "$HOME" && hermes chat --profile devops -- "$@"` |
| macOS 安装器 | `code/install.sh` | `set -e`，按 `$OSTYPE` 分支 |
| Windows 安装器 | `code/scripts/install-windows.ps1` | 装 WSL2 → WSL 内装 Hermes → 调 `install.sh` → 建快捷方式 |

### 2.3 安装流程（install.sh）

```
detect_platform($OSTYPE)
   ├─ macos   → install_macos
   │             ├─ 校验 brew，安装 yt-dlp/ffmpeg/python3/git/curl
   │             ├─ pip 安装 python 依赖
   │             ├─ 写入 ~/bin/nova（薄封装）
   │             └─ 校验 ~/.hermes 是否存在  ✗→提示安装 Hermes 并退出
   ├─ windows → show_windows_guide（仅打印 WSL2 指引）
   └─ other   → 报错退出

入口参数 --dmg：先 install_macos，再 package_dmg（hdiutil 打包 code/ 目录）
```

### 2.4 错误处理

- 缺 Homebrew / 缺 Hermes：打印指引并以非零码退出（`set -e` 保证不继续）。
- Windows 直接运行 `install.sh` 仅给指引，真实部署由 `.ps1` 在 WSL 内完成。

### 2.5 设计要点

CLI 不做业务逻辑，仅透传给 Hermes，保证「一句话终端」语义；所有智能在 M2 及以下完成。

## 3. M2·意图解析与路由模块

### 3.1 职责

把自然语言指令解析为结构化 **Intent**，并据此生成 **Plan**（Skill 编排计划）。

### 3.2 内部结构

```
NL 指令 ──> [预处理] ──> [意图解析 LLM] ──> Intent
                                   │
                                   ▼
                        [Skill 路由器] ──查询──> Skill 注册表(M4)
                                   │
                                   ▼
                        [Plan 生成器] ──> Plan{steps[], 依赖图}
```

### 3.3 处理流程

1. **预处理**：去噪、补充会话上下文（指代消解，如「那份文档」→ 具体 token）。
2. **意图解析**：调用 M5 的 LLM，按受控 schema 输出 `Intent{platform, action, output, trigger, confidence}`。
3. **置信度判定**：`confidence < 阈值(0.6)` → 回问用户澄清；否则进入路由。
4. **Skill 路由**：按 `platform × action` 在注册表中匹配候选 Skill；一个复杂需求可命中 3–5 个 Skill。
5. **Plan 生成**：确定 Skill 执行顺序与依赖（如「先采集 → 再入库 → 再通知」），标注重试/超时。

### 3.4 接口

```
parseIntent(text: string, ctx: Context): Intent
route(intent: Intent): SkillCandidate[]
plan(intent: Intent, candidates: SkillCandidate[]): Plan
```

### 3.5 关键逻辑：意图歧义处理

| 情况 | 处理 |
|------|------|
| 平台缺失（「汇总文档」未说平台） | 用上下文默认平台；无则回问 |
| 动作多义 | 取置信度最高动作，必要时确认 |
| 含触发词（每天/每小时/当…时） | 标记 `trigger`，转交 M6 注册而非立即执行 |

### 3.6 错误处理

解析失败或置信度过低 → 返回 `INPUT_INTENT_UNCLEAR`，由 M1 提示用户补充，不进入执行。

## 4. M3·编排调度与记忆模块

### 4.1 职责

按 Plan 的依赖图调度 Skill 执行，管理上下文/中间结果，处理重试、超时与结果聚合。

### 4.2 内部结构

```
Plan ──> [拓扑排序] ──> [执行器循环] ──┬─> 调用 Skill(M4)
                            │          │
              [记忆读写] <──┘          ├─> 结果写回 Context
                                       └─> 重试/超时/降级控制
                                            │
                                            ▼
                                     [结果聚合器] ──> 输出
```

### 4.3 执行流程

```
1. 拓扑排序 Plan.steps（按 dependsOn 解依赖，得执行批次）
2. for 每批次（可并行的 step 并发执行）:
     for step in 批次:
        a. 组装入参（从 Context 注入上游产物）
        b. 调用 Skill（M4），带 timeoutSec
        c. ✓ 成功 → data/artifacts 写回 Context
           ✗ 失败 → 按 error.retryable 退避重试 ≤ retry 次
                     超限 → 触发降级策略或标记失败
3. 聚合各 step 产物 → 生成最终输出（终端/飞书/文件）
4. 写执行历史（意图、Plan、各步耗时与状态）
```

### 4.4 记忆层

| 类型 | 生命周期 | 用途 |
|------|---------|------|
| 会话上下文 | 单次任务 | 步骤间传递中间结果（文档内容、采集数据…） |
| 用户偏好 | 跨会话持久 | 默认平台、收件人、输出格式 |
| 执行历史 | 持久 | 审计、失败回溯、定时任务追踪 |

> 记忆持久化复用 Hermes 能力；Nova 约定上述三类命名空间与读写时机。

### 4.5 并发与隔离

- 同一 Plan 内无依赖的 step 并发执行；有依赖的串行。
- 定时/事件任务各自独立 Session（见 M6），上下文互不污染。

### 4.6 错误处理与降级

| 场景 | 降级策略 |
|------|---------|
| GUI 自动化失败 | 改走浏览器或平台 API（若该能力存在替代 Skill） |
| 某 Skill 连续失败 | 跳过并在结果中标注「部分完成」，保留已成功产物 |
| 整链路超时 | 终止后续步，输出已完成部分 + 失败原因 |

## 5. M4·Skill 系统模块

### 5.1 职责

定义 Skill 规范，提供注册发现、分组与生命周期管理。Skill 是 Nova 的能力载体（28 个预制，4 大分组）。

### 5.2 Skill 规范（SKILL.md）

每个 Skill 自包含，目录内含描述与可执行逻辑：

```
skills/<name>/
├── SKILL.md        # 元信息：名称/描述/触发条件/输入输出/超时
└── （脚本/资源）    # 实现
```

`SKILL.md` 约定字段：`name`、`description`、`inputs`、`outputs`、`errors`、`timeout`、`group`。

### 5.3 注册与发现

```
启动 → 扫描 skills 目录 → 解析各 SKILL.md → 构建注册表
注册表索引: byName / byPlatform / byGroup / byAction
M2 路由器即查询此注册表
```

放入符合规范的新 Skill 即自动注册，无需改核心代码（即插即用）。

### 5.4 分组（28 个）

| 分组 | 数量 | Skill |
|------|------|-------|
| 🏢 办公自动化 | 13 | feishu-doc-create, google-workspace, himalaya, notion, airtable, linear, obsidian, apple-notes, apple-reminders, powerpoint, nano-pdf, ocr-and-documents, qwenvl-ocr |
| 🕷️ 数据采集 | 5 | agent-reach, browser-act, china-web-data-collection, wenshu-api-crawl, blogwatcher |
| ⚡ 自动化引擎 | 4 | cronjob, webhook-subscriptions, macos-computer-use, macos-browser-automation |
| 📤 内容发布 | 2 | wechat-article-formatter, post-anywhere |

### 5.5 生命周期

```
注册 → 被路由命中 → 校验入参 → 执行（可调用 M5/M7/M8）→ 返回标准出参 → 释放
```

### 5.6 错误处理

Skill 内部错误统一映射到 §1.2 错误码后返回，由 M3 决定重试/降级，Skill 自身不做跨步决策。

## 6. M5·AI 引擎模块

### 6.1 职责

提供 LLM 推理、视觉理解（OCR）、语音合成（TTS）与网络代理，并实现**模型路由**与故障降级。

### 6.2 能力与实现

| 能力 | 实现 | 用途 |
|------|------|------|
| LLM 推理 | DeepSeek V4（主）/ MiniMax（快）/ Qwen（容灾） | 意图解析、决策、内容生成 |
| 视觉理解 | QwenVL | 图片/截图/PDF 的 OCR 与理解 |
| 语音合成 | Edge TTS / OpenAI TTS | 文本转语音输出 |
| 网络代理 | HTTP / SOCKS5 代理层 | 适配国内外网络环境 |

### 6.3 模型路由策略

```
infer(prompt, opts):
   任务分级:
     ├─ 简单/高频（分类、抽取、改写）   → MiniMax  （低延迟/低成本）
     ├─ 复杂推理（决策、风险判断、规划） → DeepSeek V4（主力）
     └─ 含图像                          → QwenVL
   故障降级:
     主模型超时/报错 → 退避重试 → 仍失败则切 Qwen 容灾
```

### 6.4 接口

```
infer(prompt|messages, { model?, tools?, temperature?, stream? }): Result | Stream
vision(image, prompt): Result
tts(text, voice): AudioRef
withProxy(request): request    // 出站统一经代理层
```

### 6.5 关键设计

- **模型选择对上层透明**：M2/M4 只声明任务类型，由 M5 决定具体模型。
- **流式输出**：长文本生成支持流式，改善 CLI 交互体验。
- 与传统 RPA 的本质差异在此层：处理非确定性、非结构化输入。

### 6.6 错误处理

模型层错误归类为 `RATE_*`/`NET_*`/`RUNTIME_*`；连续失败触发容灾切换，仍失败上报 `FATAL_MODEL`。

## 7. M6·Runtime 调度模块

### 7.1 职责

将一次性指令变为长期运行的自动化：定时（cronjob）与事件（webhook）触发，保证任务隔离、重试、超时与日志。

### 7.2 两类触发器

| 机制 | 触发方式 | 注册来源 | 典型场景 |
|------|---------|---------|---------|
| cronjob | 时间（分/时/天/周） | 含触发词的指令经 M2 标记 | 日报、周报、定期巡检 |
| webhook | 事件（表单提交/回调/push） | 显式订阅 | 审批流、表单入库、CI 通知 |

### 7.3 注册与执行流程

```
注册:
  M3.register(trigger, instruction)
    → cronjob: 写入 Hermes cron（schedule + sessionTemplate）
    → webhook: 注册订阅端点 + 事件过滤条件

触发执行:
  到点 / 收到事件
    → 拉起独立 Session（注入预设指令与必要上下文）
    → 进入主链路（M2→M3→M4…）
    → 记录执行日志；失败按策略重试；超时中止
```

### 7.4 隔离与可靠性

- 每个定时/事件任务独立 Session，互不干扰。
- 支持失败重试、超时控制、执行历史留存。
- webhook 事件去重，避免重复触发。

### 7.5 错误处理

触发但执行失败 → 记录失败原因，按配置重试；多次失败可推送告警（复用集成层通知能力）。

## 8. M7·自动化执行模块

### 8.1 职责

承载「动手操作」类执行：macOS 桌面 GUI 自动化与浏览器自动化。

### 8.2 两类执行器

| 执行器 | 底层 | 能力 | 对应 Skill |
|--------|------|------|-----------|
| 桌面 GUI | macOS Accessibility API | 操作任意 GUI 软件（Excel/微信/PS/Keynote…），无需插件 | macos-computer-use |
| 浏览器 | CDP / Playwright | JS 渲染页面采集、表单填写、滚动加载 | macos-browser-automation, browser-act |

### 8.3 自然语言定位（核心差异）

```
传统 RPA: 选择器(第3行第5列/CSS 选择器) → 易失效
Nova:     自然语言描述「那个蓝色提交按钮」
            → AI 结合可访问性树/DOM/视觉定位元素
            → 失效时重新定位（自愈），而非中断
```

### 8.4 接口

```
act(target, op):
  target = { app | url, locator: NL|selector }
  op     = { action: click|type|read|scroll|screenshot, value? }
returns { ok, data?, screenshotRef? }
```

### 8.5 处理流程（浏览器采集示例）

```
打开页面 → 等待渲染（含 AJAX/滚动加载）
  → AI 识别目标区域 → 提取字段
  → 翻页/滚动循环直到结束条件
  → 结构化输出（交 M8 入库或 M3 聚合）
```

### 8.6 错误处理

- 元素定位失败：重试 + AI 重新定位；仍失败标记 `RUNTIME_LOCATE_FAIL`，由 M3 决定降级。
- 反爬/限流：经 M5 代理层 + 限速；策略见 `china-web-data-collection`。

## 9. M8·集成适配模块

### 9.1 职责

统一封装各目标平台的认证、读写与发布，屏蔽差异，使「数据从任意入口进、任意目标出」。

### 9.2 适配范围

| 类别 | 平台 | 主要 Skill |
|------|------|-----------|
| 协作办公 | 飞书 / 企业微信 / 钉钉 | feishu-doc-create |
| 国际生态 | Google Workspace / Notion / Airtable / Linear | google-workspace, notion, airtable, linear |
| 邮件 | IMAP/SMTP | himalaya |
| 文档 | PPT / PDF / OCR | powerpoint, nano-pdf, ocr-and-documents |
| 发布 | 公众号 / 小红书 / 视频号 / Twitter / 知乎 | wechat-article-formatter, post-anywhere |

### 9.3 统一适配接口

```
auth(platform): Credential          // 复用 Hermes 凭证存储
read(platform, resource, query): Data
write(platform, resource, payload): Ref
publish(platform, content): Ref     // 发布类专用
```

### 9.4 设计要点

- **最小权限**：仅申请任务所需的平台权限范围。
- **凭证集中**：认证信息由 Hermes 安全存储，适配层按需取用，不落地明文。
- **单元化**：每个平台一个 Skill，平台 API 变更只影响对应 Skill，升级面可控。

### 9.5 错误处理

认证失效 → `AUTH_EXPIRED` 触发重新授权；平台限流 → `RATE_*` 退避重试；字段不符 → `INPUT_*` 回问或用默认值。
